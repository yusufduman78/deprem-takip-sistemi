1.1  Sensor Station Firmware & MPU6050 Interface
SRS-1: Software Scope and Boundaries
This SRS covers only the software responsibilities of the Sensor Station Hardware Lead. Wi-Fi connection, MQTT, LCD menu, or alarm management coded by other individuals are outside the scope of this section.
Software components:
MPU6050 register-level initialization and data reading
Interrupt-based motion detection via the INT pin
Sensor-microcontroller communication verification tests

SRS-3: Register-Level Coding Requirements
No external libraries shall be used directly. All read/write operations must be performed manually using Wire.beginTransmission(), Wire.write(), and Wire.requestFrom().
Initialization sequence:
Wake the sensor from sleep mode by writing 0x00 to PWR_MGMT_1.
Set the sampling rate to 125 Hz via SMPLRT_DIV.
Enable the internal Digital Low-Pass Filter (DLPF) via the CONFIG register.
Select the ±8g range using ACCEL_CONFIG.
Write MOT_THR and MOT_DUR registers; activate motion interrupt by writing 0x40 to INT_ENABLE.
SRS-4: Interrupt-Based Motion Detection
The MPU6050 compares the acceleration value against the MOT_THR threshold. When exceeded for the duration of MOT_DUR, it pulls the INT pin HIGH. The ISR function shall be kept as short as possible — only setting volatile bool motionFlag = true and reading INT_STATUS (0x3A) to clear the flag. I2C reading, Serial print, or Wi-Fi operations shall not be performed inside the ISR.
SRS-5: Communication Verification Tests
Test 1 — I2C Connection: Run I2C Scanner sketch; verify 0x68 address is listed.
Test 2 — Raw Data Integrity: While stationary, ACCEL_Z ≈ +4096 (±8g). 100 samples read, standard deviation calculated.
Test 3 — INT Pin Interrupt: Lightly tap sensor; ISR must trigger with ≤ 50 ms delay. Zero false positives in quiet environment for 10 minutes.

1.2  Signal Processing & Firmware DSP Algorithms
2.1 System Initialization and Register Mapping
Upon boot, the firmware shall bypass high-level wrapper libraries and directly configure the MPU6050 registers via the I2C bus at a clock speed of 400 kHz.
REQ-SRS-01 (Wake-up): Write 0x00 to PWR_MGMT_1 register (0x6B) to disable sleep mode.
REQ-SRS-02 (Sensitivity): Configure ACCEL_CONFIG register (0x1C) to 0x10, setting full-scale range to ±8g (4096 LSB/g).
REQ-SRS-03 (Hardware DSP): Enable on-chip Digital Low-Pass Filter by writing 0x04 to CONFIG register (0x1A), capping bandwidth at 21 Hz.
REQ-SRS-04 (Interrupt Configuration): Write 0x01 to INT_ENABLE register (0x38) to assert the physical INT pin whenever new data is written to the internal FIFO buffer.
2.2 Interrupt-Driven Data Acquisition
Polling is strictly prohibited to maximize CPU efficiency and avoid blocking the main task scheduler.
REQ-SRS-05 (ISR Logic): An ISR shall be attached to GPIO14 on a RISING edge. The ISR shall contain minimal logic — strictly limited to setting a volatile bool dataReady = true flag.
REQ-SRS-06 (Burst Read): When the main loop detects the dataReady flag, the software shall execute a 6-byte burst read starting from register 0x3B (ACCEL_XOUT_H) to fetch X, Y, and Z axes atomically.
2.3 Digital Signal Processing (DSP) Pipeline
REQ-SRS-07 (Static Calibration): During the first 5 seconds, average 500 samples to compute static gravitational offset (1g on Z-axis, 0g on X and Y). Subtract offset from all subsequent readings.
REQ-SRS-08 (Software Filtering): Apply a First-Order IIR Low-Pass Filter to calibrated data:
yi[n] = α · xi[n] + (1−α) · yi[n−1]  (α = smoothing coefficient)
REQ-SRS-09 (Vector Synthesis): Calculate Peak Ground Acceleration (PGA) as Euclidean norm: Atotal = √(ax² + ay² + az²)
2.4 Magnitude Estimation and Trigger Logic
REQ-SRS-10 (Richter Conversion): Estimate local magnitude M using: M ≈ log10(PGA) + C  where C = 3.0 baseline calibration constant, PGA in cm/s².
REQ-SRS-11 (Event State Machine): If calculated magnitude M exceeds trigger threshold (e.g. 3.5), transition to ALARM state.
REQ-SRS-12 (Debouncing): Upon triggering, enforce a 45-second lockout period where subsequent peaks are recorded as the same event.
REQ-SRS-13 (Payload Generation): In ALARM state, construct a structured JSON payload containing trigger timestamp, max PGA, and calculated Richter magnitude, forwarding it to the backend module.
— Talha Yüncü
1.3  Main Station UI & Interaction
The software logic for the Main Station prioritizes responsiveness and ease of use, ensuring that the system remains accessible even during high-stress situations.
Functional Requirements (FR)
FR1 — User Interface (UI) Menu System: The system must provide a hierarchical menu on the 16x2 LCD. Options: 1. Last Event, 2. System Status, 3. Silence Alarm, 4. Sensitivity Settings.
FR2 — Hardware Interrupt Integration: The touch keypad must be handled via External Interrupts (ISR) rather than polling. This ensures that a user's touch is detected instantly, even if the processor is busy updating the display or communicating with the cloud.
FR3 — Software Debouncing: Implement a timing-based debouncing algorithm (typically 50 ms–200 ms) to prevent multiple 'ghost' triggers from a single touch on the capacitive keypad.
FR4 — Dynamic Alerting: The system must trigger the Buzzer using PWM to create a rising/falling siren effect during an earthquake event.
FR5 — Visual Feedback: The RGB LED must change states based on the system mode (Blinking Red for Alert, Solid Blue for Cloud Syncing).
Non-Functional Requirements (NFR)
NFR1 — Latency: The delay between a keypad touch and a menu update must not exceed 100 ms.
NFR2 — Clarity: Text on the LCD must be formatted for high legibility, using clear headers and concise status updates.
NFR3 — Reliability: The ISR must be kept 'lean' (minimal code inside) to prevent the watchdog timer from resetting the NodeMCU.
Implementation Logic (Technical Workflow)
Interrupt Handling Strategy:
The keypad's IRQ pin is tied to a hardware-capable GPIO. Using the IRAM_ATTR attribute ensures the code runs from RAM, providing the fastest possible response time.
void IRAM_ATTR handleKeypress() {
  keyPressed = true;  // Flag for main loop to process the menu change
}
Menu Navigation Logic:
The menu is managed using a 'State Machine' approach. Depending on the MenuState variable, the LCD displays different strings and the keypad inputs are mapped to different functions (e.g., in 'Alarm Mode,' any key acts as a 'Mute' button).
PWM for Audio:
The buzzer frequency is varied to increase urgency of the warning:  f(t) = fbase + A · sin(2π · fmod · t). This creates a 'warble' effect that is much more likely to wake a sleeping user than a flat tone.

1.5  IoT Communication & Data Bridge Layer
Functional Requirements
The IoT Communication and Data Bridge Layer shall provide the following functional capabilities:
Data Reception from Sensor Station: The layer shall receive seismic data packets sent from the Sensor Station in JSON format via MQTT protocol.
Real-time Data Forwarding: Upon receiving a data packet, the layer shall instantly forward the same packet to: (a) the Main Control & Alert Station for local alarm activation, and (b) the Firebase Cloud platform for mobile/web notifications and statistical logging.
Earthquake Alarm Prioritization: When the 'DEPREM' flag is set in the incoming packet, the layer shall transmit this alarm message with the highest priority to ensure minimum latency for both local buzzer/LED activation and cloud-based push notifications.
Bidirectional Communication Support: The layer shall support bidirectional messaging, allowing the Main Station to send commands (e.g., 'acknowledge', 'silence alarm', 'request historical data') back to the Sensor Station when necessary.
Scalability for Multiple Sensors: The communication structure shall be designed to support future expansion, allowing multiple sensor stations to publish data to the same MQTT broker without requiring major code changes.
Cloud Integration: The layer shall publish earthquake intensity (PGA/Richter value), timestamp, and alarm status to Firebase Realtime Database for real-time dashboard visualization and e-mail/SMS notification services.

1.6  Dashboard & Global Alert System
Overview
This layer forms the user-facing interface of the Earthquake Monitoring and Warning System. It reads earthquake data written to Firebase Realtime Database by the IoT Communication Layer in real time, visualizes the data on a responsive web/mobile dashboard, and delivers immediate push notifications and email alerts to users whenever an earthquake is detected (deprem_flag = true).
No physical hardware, sensors, or microcontrollers are used in this layer. It operates entirely on cloud services (Firebase) and end-user devices (smartphones, tablets, and web browsers).
Integration with IoT Layer — Expected JSON payload structure:
{ "device_id": "SENSOR_01", "timestamp": "2026-03-26T15:30:45",
  "pga": 0.85, "richter": 4.2, "deprem_flag": true,
  "accel_x": 0.45, "accel_y": -0.12, "accel_z": 9.81 }
This layer never writes data back to Firebase. All UI updates and alerts are triggered by changes in the database.
Functional Requirements
FR 1.1 — Real-Time Data Visualization:
Listen to the /earthquake_events path in real time using Firebase SDK onValue() and update the UI instantly upon new events.
Main dashboard displays: system status (NORMAL / EARTHQUAKE ALARM), latest Richter value, PGA value, and event timestamp.
Live accelerometer graph (accel_x, accel_y, accel_z) plotted over time.
When deprem_flag = true, the interface switches to alarm mode with color change and visual alert banner.
FR 1.2 — Historical Data & Statistics:
Display past earthquake events in list or card view.
Support filtering by date range and minimum Richter value.
Provide bar charts and line charts for Richter and PGA values over time.
Show summary statistics: total number of events, maximum magnitude, and 7-day average.
FR 1.3 — Notifications (when deprem_flag = true):
Send push notifications via Firebase Cloud Messaging (FCM) to all subscribed devices.
Simultaneously send an email alert via Firebase Cloud Functions + Nodemailer (Gmail SMTP or similar).
Notification content example: "EARTHQUAKE ALERT! Richter: X.X | PGA: Y.Y cm/s² | Time: HH:MM"
Users can configure notification preferences (push on/off, email on/off, email address). Preferences are saved in Firebase.
FR 1.4 — User Interface:
Tabs / Pages: 'Live Monitor', 'Event History', 'Statistics', and 'Notification Settings'.
Fully responsive design that works seamlessly on desktop and mobile browsers.
Installable as a Progressive Web App (PWA) for home-screen app-like experience on iOS and Android.
Dark / Light theme toggle support.

HRD-3: I2C Connection Plan
Although pull-up resistors are already present on the GY-521 module, it may be necessary to add external 4.7 kΩ resistors when using long cables or in noisy environments.
HRD-4: Noise Filtering and Bypass Capacitors
Since the power pins of the MPU6050 perform sensitive analog measurements, high-frequency noise caused by digital switching degrades measurement accuracy.
A 100 nF ceramic capacitor shall be soldered between the VCC and GND pins of the MPU6050, as close to the pins as possible.
A 10 µF electrolytic capacitor suppresses low-frequency oscillations along the power rails.
I2C lines (SCL/SDA) must be kept shorter than 50 cm and routed away from parallel power lines.
HRD-5: Physical Mounting Requirements
Surface: The MPU6050 must be mounted on a rigid surface. A freely swinging module measures incorrect vibrations.
Axis alignment: The arrow mark on the GY-521 indicates the 'North' direction. This direction must be noted during installation; Z-axis aligned with gravity ≈ 1g.
Soldering: The bypass capacitor should be soldered as close to the base of the pin as possible; I2C lines must not be run in parallel.
— Berkant Arslan
2.2  Signal Processing Hardware Interface
1.1 Core Hardware Components
Primary Microcontroller (MCU1): NodeMCU V3 (ESP8266 ESP-12E module).
Seismic Sensor: MPU6050 (6-Axis IMU with internal 16-bit ADCs).
Communication Bus: Two-wire Inter-Integrated Circuit (I2C).
1.2 Electrical and Pin Routing Specifications
Power Supply Routing: The MPU6050 VCC pin shall be powered directly from the ESP8266's regulated 3.3V output rail. Both devices share a common Ground (GND).
I2C Data Line (SDA): The sensor's SDA pin shall be routed to NodeMCU Pin D2 (GPIO4).
I2C Clock Line (SCL): The sensor's SCL pin shall be routed to NodeMCU Pin D1 (GPIO5).
Hardware Interrupt Line (INT): The sensor's INT pin shall be physically wired to NodeMCU Pin D5 (GPIO14).
Pull-up Requirements: If the MPU6050 breakout board lacks internal pull-up resistors for the I2C lines, external 4.7 kΩ resistors must be placed between the 3.3V rail and the SDA/SCL lines.
1.3 Operational Hardware Constraints
The MPU6050 AD0 pin must be physically grounded to lock the I2C address to 0x68.
The physical distance between the ESP8266 and the MPU6050 shall not exceed 15 cm to prevent parasitic capacitance from degrading the 400 kHz I2C signal integrity.
— Talha Yüncü
2.3  Main Station Hardware & User Interaction
The Main Station serves as the central hub where data is visualized and user commands are input. This section details the physical assembly and wiring of the interface components.
1.1 Core Interface Components
Microcontroller: NodeMCU V3 (ESP8266 ESP-12E). Role: Managing the I2C display bus, processing keypad interrupts, and driving alert peripherals.
Display: 16x2 LCD with I2C Interface. Connectivity: Connected via SDA (GPIO 4) and SCL (GPIO 5). This reduces pin usage and simplifies wiring.
Input: 16-Channel Capacitive Touch Keypad. Role: Navigating the system menu and acknowledging alarms.
Alert Peripherals:
KY-006 Passive Buzzer: Used for generating varied frequency sirens.
5mm RGB LED: Provides visual status (e.g., Green for 'Safe,' Red for 'Alarm').
Prototyping & Final Assembly:
Breadboard: Used for initial circuit verification and signal testing.
Pertinaks (Perfboard): The final circuit is soldered onto a perforated copper board to ensure vibration resistance and long-term durability.
1.2 Physical Integration & Protection
Current Limiting: 220 Ω resistors are integrated into the RGB LED circuit to prevent overcurrent from the NodeMCU GPIO pins.
Soldering Standards: All connections on the pertinaks board are reinforced with high-quality solder joints to prevent 'cold joints' that could fail during an actual seismic event.

The built-in Wi-Fi antenna on each NodeMCU is sufficient for home or laboratory environments; no external antenna is required.
